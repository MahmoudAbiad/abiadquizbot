# تحديث #2 — تتبّع شامل للأسئلة + شحن برسالة + الطلاب النشطون

## 1) الملفات المعدّلة (استبدلها بنفس الأسماء)
| الملف | التعديل |
|---|---|
| `supabase_helper.py` | 3 دوال جديدة: `log_quiz_response`, `admin_get_users_question_totals`, `admin_get_active_users` |
| `execution.py` | معالج إجابة الـ Poll (`handle_poll_answer`) أصبح يسجّل كل سؤال بجدول `quiz_responses` |
| `admin.py` | لوحة `/fetchall` + بحث الطالب + تصدير CSV تعرض الآن "الأسئلة المحلولة" الشاملة + زر شحن مع رسالة + زر الطلاب النشطين + **إصلاح خطأ برمجي** كان سيكسر البرودكاست |
| `keyboards.py` | مزامنة نفس الأزرار الجديدة بالنسخة المكررة (غير مستخدمة فعلياً لكن حفاظاً على تناسق الملف) |

## 2) لماذا كانت أسئلة المشاركة/المفضلة "ناقصة"؟
`quiz_attempts` كانت أصلاً تسجّل كل محاولة كويز (ملف/نص/مشترك/مفضلة/كاش) — هذا الجزء كان يعمل. لكن:
- عمود `users.total_questions` يمثّل **فقط** الأسئلة المدفوعة عند التوليد من ملف/نص (منطقي، لأنه مرتبط بالفوترة) — وهذا هو الرقم الوحيد الذي كانت لوحة الإدارة (`/fetchall`، بحث الطالب، CSV) تعرضه، فكان يبدو وكأن أسئلة المشاركة/المفضلة "غير محسوبة".
- جدول `quiz_responses` (لتسجيل كل سؤال بمفرده: صح/خطأ) كان **موجوداً بقاعدة البيانات لكن فارغاً تماماً (0 صف)** — لا شيء كان يكتب فيه إطلاقاً، بغض النظر عن مصدر الكويز.

## 3) ما الذي تغيّر فعلياً؟
- **`quiz_responses` أصبح يُملأ تلقائياً** الآن من نفس المعالج المشترك بين كل أنواع الكويزات، فيشمل تلقائياً: ملف، نص، كاش، **مشترك**، **مفضلة**، وحتى تجربة الإدارة. لا حاجة لأي تفريق يدوي بين المصادر — لأن كودها موحّد أصلاً.
- **لوحة `/fetchall` وبحث الطالب وتصدير CSV** أصبحت تعرض الآن سطرين منفصلين وواضحين لكل طالب:
  - `📝 أسئلة مولّدة (مدفوعة من ملفات/نصوص)` ← نفس الرقم القديم (`total_questions`)، محفوظ كما هو لأنه مرتبط بالفوترة.
  - `🧾 إجمالي الأسئلة التي حلّها (كل المصادر)` ← رقم جديد من `quiz_attempts`، ويتفرّع إلى: ملفات/كاش | مشتركة | مفضلة.
- ملف الـ CSV المُصدَّر أصبح فيه أعمدة منفصلة: `Generated Questions (Paid)`, `Practiced Questions Total`, `Practiced From Files/Cache`, `Practiced From Shared`, `Practiced From Favorites`.

## 4) 💬 شحن نقاط + رسالة مخصصة
من `/admin` → بحث عن طالب → **💰 شحن نقاط للمستخدم** → زر جديد **💬 شحن نقاط + رسالة مخصصة للطالب**:
1. يسألك عن عدد النقاط.
2. يسألك عن نص الرسالة (أي نص حر).
3. يشحن النقاط فعلياً، ثم يرسل للطالب إشعار الشحن المعتاد **مع رسالتك مضمّنة بداخله** في قسم "💬 رسالة من الإدارة".
الشحن السريع (+10/+50/+100) والشحن اليدوي بدون رسالة ما زالا يعملان كما كانا تماماً.

## 5) 🟢 الطلاب النشطون الآن
من `/admin` → زر **🟢 الطلاب النشطون الآن** → اختر النافذة الزمنية: ساعة / 3 / 6 / 12 / 24 ساعة.
لكل طالب نشط بهذه الفترة تُعرض: الاسم الكامل، اليوزر، الآيدي، آخر نشاط قام به (نوع الحدث + وقته)، وعدد الأحداث التي سجّلها خلال الفترة. القائمة مبنية على جدول `usage_events` الموجود أصلاً (لا حاجة لأي جدول جديد).

## 6) لا حاجة لأي Migration جديد
كل الجداول المطلوبة (`quiz_attempts`, `usage_events`, `quiz_responses`) كانت موجودة بالفعل. لم يُضَف أي عمود أو جدول جديد — فقط تفعيل استخدام `quiz_responses` وإضافة استعلامات تجميع جديدة.
مرفق ملف **اختياري** `optional_performance_indexes.sql` (فهارس فقط، لا يمس أي بيانات) لتسريع هذه الاستعلامات مع نمو عدد الطلاب — شغّله من Supabase SQL Editor إذا رغبت.

## 7) ⚠️ ملاحظة أمنية مهمة (غير متعلقة بطلبك، لكن يستحق الانتباه)
أثناء فحص قاعدة البيانات لاحظت أن **13 جدولاً بما فيها `users`, `quizzes`, `usage_events`, `quiz_attempts`... كلها بدون Row Level Security (RLS) مفعّل**. عملياً هذا يعني: أي شخص يملك الـ anon key الخاص بمشروعك (وهو غالباً مضمّن بأي كود Frontend لو استُخدم) يستطيع نظرياً قراءة/تعديل/حذف أي صف بأي جدول مباشرة عبر REST API الخاص بـ Supabase، بغض النظر عن كود البوت. بما إن بوتك Backend فقط (لا Frontend يستخدم anon key)، الخطر الفعلي محدود طالما مفتاح الـ Service/anon لم يتسرّب — لكن يستحق المراجعة. لم أطبّق أي تعديل هنا لأن تفعيل RLS بدون سياسات (Policies) صحيحة سيقطع وصول البوت نفسه لقاعدة البيانات. إذا رغبت أن أساعدك ببناء سياسات RLS مناسبة لاحقاً أخبرني.
