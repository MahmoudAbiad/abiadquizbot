# نظام تتبع استخدام الطلاب — دليل التركيب

## 1) خطوة أولى إلزامية: تشغيل SQL Migration
افتح **Supabase → SQL Editor** وشغّل ملف `migration_usage_analytics.sql` كاملاً.
ينشئ جدولين جديدين فقط (`usage_events`, `quiz_attempts`) ولا يلمس أي جدول موجود — آمن للتشغيل على قاعدة بياناتك الحالية مباشرة.

## 2) الملفات المعدّلة
استبدل الملفات التالية بنفس الأسماء في مشروعك:

| الملف | التعديل |
|---|---|
| `supabase_helper.py` | إضافة قسم "Usage Analytics & Tracking" بالكامل في آخر الملف (دوال التسجيل + دوال التجميع للأدمن) |
| `start.py` | تسجيل حدث `bot_start` عند كل `/start`، وحدث `shared_link_opened` عند الروابط العميقة |
| `files.py` | تسجيل رفع المحتوى (`content_uploaded`)، طلب التوليد (`quiz_generation_requested`)، التوليد الفعلي (`quiz_generated`)، واستخدام الكاش (`cached_quiz_used`) |
| `execution.py` | **الجزء الأهم**: إنشاء سجل `quiz_attempts` عند بدء كل كويز، وإغلاقه عند الإكمال أو الإيقاف المبكر، بالإضافة لتسجيل المشاركة/الحفظ/التقييم/الملاحظات |
| `sharing.py` | تسجيل إنشاء وفتح روابط المشاركة |
| `leaderboard.py` | تسجيل نشر النتيجة وعرض لوحة الشرف |
| `admin.py` | قسم "📈 تحليلات الاستخدام" الكامل بلوحة الأدمن + عرض نشاط طالب محدد + تصدير CSV لسجل الأحداث |
| `keyboards.py` | مزامنة نفس الأزرار الجديدة في النسخة المكررة من لوحات الأدمن |

## 3) ما الذي يتتبعه النظام؟
- **usage_events**: كل حدث (بدء بوت، رفع ملف/صورة/نص، توليد كويز، مشاركة، حفظ، تقييم، ملاحظة...) مع `metadata` بصيغة JSON لأي تفاصيل إضافية.
- **quiz_attempts**: كل محاولة كويز على حدة — المصدر (ملف/صورة/ألبوم/نص/مشترك/مفضلة/كاش)، عدد الأسئلة، النتيجة، هل اكتمل أم توقف الطالب مبكراً، ومدة الحل بالثواني.

كل دوال التسجيل مصممة لتكون **صامتة عند الفشل** (try/except يسجل بالـ logger فقط) حتى لا يتأثر تدفق الطالب أبداً إذا حدث خطأ بقاعدة البيانات.

## 4) ما تراه بلوحة الأدمن
اضغط `/admin` → **📈 تحليلات الاستخدام**:
- مستخدمون نشطون خلال 7/30/90 يوم (أزرار تبديل)
- عدد محاولات الكويز ومعدل الإكمال
- متوسط مدة الحل ومتوسط النتائج
- الأحداث الأكثر تكراراً + توزيع مصادر الكويزات (ملف/صورة/نص...)
- **📅 النشاط اليومي**: رسم نصي بسيط (Bar Chart) لآخر 14 يوم
- **📥 تصدير CSV** لسجل الأحداث الخام (حتى 5000 حدث)

وعند البحث عن أي طالب، زر جديد **📈 نشاط هذا الطالب** يعرض آخر محاولاته وأحداثه بالتفصيل.

## 5) الأداء — لا تأخير إطلاقاً على الطالب
كل عمليات التتبع **بدون استثناء** لا تنتظر قاعدة البيانات قبل أن يكمل البوت رده للطالب:

- `log_usage_event`, `complete_quiz_attempt`, `mark_quiz_attempt_stopped` → تُشغَّل دائماً عبر `asyncio.create_task(...)` (إطلاق بالخلفية وعدم الانتظار).
- `start_quiz_attempt` (عند بدء كل كويز) → أصبحت دالة **غير معطّلة إطلاقاً (Zero-latency)**: تُنشئ معرّف تتبع (`client_ref`) فوراً بجانب البوت دون أي اتصال بقاعدة البيانات، وتُطلق عملية الإدراج الفعلية بمهمة خلفية منفصلة. الطالب يشوف السؤال الأول فوراً دون انتظار ولو جزء من الثانية.
- إذا فشل أي استدعاء تتبع لأي سبب (انقطاع اتصال، خطأ بقاعدة البيانات...) فهو يُسجَّل بالـ logger فقط ولا يرمي استثناء أبداً — لا يمكن لنظام التتبع أن يُفشل أو يؤخر أي نشاط للطالب.

> ملاحظة تقنية بسيطة: بما إن إدراج بداية المحاولة يحصل بالخلفية، هناك احتمال نظري ضئيل جداً (كويز سؤال واحد يُجاب خلال أجزاء من الثانية) أن يُغلق الكويز قبل إتمام إدراج بدايته. هذا لا يؤثر على تجربة الطالب أبداً، وأقصى تأثير محتمل هو عدم تسجيل تلك المحاولة النادرة بالإحصائيات — وهذا ثمن مقبول جداً مقابل ضمان عدم تأخير أي طالب أبداً.
